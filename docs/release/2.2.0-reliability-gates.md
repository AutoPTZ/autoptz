# AutoPTZ 2.2.0 Reliability Gates

This file defines the release bar for 2.2.0. The release is blocked until the
benchmarks and tests below pass on the target platforms.

Latest local fake-NDI evidence is summarized in
[`2026-06-30-local-gate-notes.md`](2026-06-30-local-gate-notes.md). That file is
not a release sign-off; it records the current model-server/unified-pose
decisions and what remains unproven.

## Product Contract

- One normal AutoPTZ mode.
- Fixed zoom by default.
- No visible tracking speed presets or normal-user PTZ tuning knobs.
- No generic normal-user Experimental Features menu. Research switches are
  dev/benchmark inputs only until evidence promotes them to automatic defaults
  or retires them.
- Persisted services must come back online at startup/restart exactly as saved;
  stale queued commands from a previous stopped session must never override the
  persisted service map.
- AutoPTZ Mark is a Labs benchmark that uses the same production pipeline as a
  realistic deployment test.
- The standalone process-per-camera/model-per-child flag is retired and ignored.
  Model-server remains only as an explicit release-gated architecture candidate
  until Mark artifacts prove it beats production without RAM/CPU cliffs.
- The center quiet-zone/deadband is an internal controller primitive, not a
  normal user tuning surface. Normal UI may draw it as a passive indicator only;
  users must not drag, resize, or select tracking speed/safe-zone controls.

## Stream Gates

### Required Mark Camera Counts

Mark must support these camera counts:

`1, 2, 4, 6, 8, 12, 16`

The 6-camera option is the mid-tier gate. The 8-camera option is the release
gate for 1080p30 source delivery. The 12- and 16-camera options remain for
headroom characterization after the 8-camera release gate is stable.

### 8-Stream Gate

Run 8 fake NDI streams at 1080p30 for 30 minutes.

Pass criteria:

- `steady_state_app_induced_drops == 0` for every step and camera. The raw
  `app_induced_drops` total may be non-zero only when the step also reports
  `source_mutation_events > 0` and the drops are accounted under
  `source_mutation_allowed_drops`.
- Delivered source fps does not collapse to 4-7 fps.
- Preview, virtual camera output, inference, and PTZ command work do not block
  capture.
- CPU and RAM remain bounded over the full run.
- Mark writes a JSON artifact with per-camera source fps, delivered fps,
  duplicate/stale frame counts, NDI buffer/conversion/copy time, app-induced
  drops, latency, CPU, and memory.
- Mark CSV exports must flatten the same per-camera source evidence needed for
  release triage: delivered/source fps, source drop estimate, app-induced drops,
  duplicate/stale frame counts, NDI FourCC, NDI buffer/conversion/copy time, and
  NDI queue depth/receiver performance counters when available. CSV must also
  expose the step-level gate fields: `step_app_induced_drops`,
  `steady_state_app_induced_drops`, `source_mutation_events`,
  `source_mutation_allowed_drops`, and `source_mutation_drop_grace_s`.
- Optional NDI SDK queue, connection, and performance-counter probes are sampled,
  not polled on every delivered frame; per-frame work stays limited to capture,
  SDK-buffer handoff, format conversion, contiguous-copy accounting, frame
  accounting, and duplicate/stale detection.

Scheduled inference skips are allowed on CPU-only systems when explicitly
reported as scheduler skips. They must not be counted as capture drops.
The only capture-drop exception is during an explicit source add/remove
transition. Those drops must be reported separately and must never be folded into
the steady-state pass/fail bucket.

## Tracking Gates

Default tracking is Simple Follow:

- Person detector plus tracker plus a target state machine.
- Valid target states: `acquire`, `track`, `hold`, `lost`, `reacquire`.
- Bounding boxes are acceptable as the universal fallback evidence, not as direct
  PTZ commands. A box must pass target-lock validation before it can refresh the
  trusted target; unusable boxes must not become the future hold/reacquire
  reference.
- Bbox-only PTZ framing must use stabilized framing evidence. When a locked box
  overlaps its prior box but changes shape sharply, the aim point and subject
  height blend against the last trusted aim instead of chasing raw bbox geometry.
- LOST or stale boxes never drive PTZ.
- Degenerate, too-small, collapsed, off-frame, held, or one-frame-teleported boxes
  never drive PTZ.
- Low-overlap target jumps require repeated consistent evidence or trusted pose/body
  evidence before PTZ reacquires.
- Detector-skip frames do not re-feed stale detections as fresh evidence.
- Lost target behavior defaults to hold/stop, not search or zoom-out.
- Auto-zoom is off by default and is not part of the 2.2 release gate.
- The internal quiet-zone/deadband must remain, or be replaced by an equivalent
  tested hysteresis/hold mechanism, because it is what prevents detector jitter
  from becoming PTZ micro-corrections. The retired experiment is the user-editable
  framing box, not the anti-bobbing deadband itself.
- Restart/resume tests must verify detection, tracking, face recognition, pose,
  and ReID service switches restore from the current persisted state, including
  after a stopped-engine reset and after returning from Mark.

Synthetic and hardware tests must cover:

- A moving person.
- Crossing people.
- Temporary occlusion.
- Reduced inference cadence.
- Delayed frames.
- PTZ motion feedback.
- No repeated command direction reversal that produces visible bobbing.
- No one-frame bbox teleport that commands a pan/tilt correction.

## Platform Gates

Record Mark JSON artifacts for:

- Apple Silicon Mac.
- Intel Mac.
- CPU-only Windows.
- Accelerator-backed Windows.

Each platform must include at least:

- 6-camera run.
- 8-camera run.
- Mark GUI lifecycle run: open, run, complete, close, return to normal AutoPTZ.
- Headless Mark JSON run.

## Mark UI and Lifecycle Gates

AutoPTZ Mark is a benchmark harness, not a second product mode. Its UI must stay
close to AutoPTZ while making release gates obvious:

- Mark's default workload is `simple_follow`: detector + tracker only. This is
  the 2.2 production baseline.
- Mark must keep comparison profiles for service-cost research: `streams`
  (capture/preview only), `simple_follow` (detector + tracker), `pose_follow`
  (adds pose aiming), and `full` (adds face recognition and ReID).
- Mark JSON output must include `profile_description` and `profile_features` so
  saved runs can prove which service stack was active when comparing default vs
  unified/pose-heavy paths.
- Mark JSON output must also include managed `experimental_flags` such as
  `AUTOPTZ_UNIFIED_POSE` and `AUTOPTZ_NDI_COLOR_FORMAT`, so unified-vs-default
  runs are comparable after the fact.
- The performance chart must label the target FPS line, current camera count,
  current minimum delivered FPS, steady-state app-induced drop status, and any
  source-mutation-allowed drops. Green/red points are not enough by themselves.
- Stop and Exit Mark must remain adjacent top-level controls. Stop is a danger
  action while the run is active; Exit Mark remains available at all times.
- Closing Mark from the window button, choosing Return/Quit, or quitting the app
  while a benchmark is in progress must stop the GUI pump, cancel the active
  sampler, wait the controller, stop the isolated Mark engine, and detach Mark's
  log handler exactly once.
- Cancellation must interrupt Mark warmup/dwell waits. Exiting mid-test must not
  block on the full current benchmark dwell window.

## Face Identity Gates

Face identity remains optional for tracking, but when the service is enabled it
must be visibly and operationally reliable:

- Windows builds must load the bundled InsightFace pack or report a clear
  Services-panel warning when the pack is missing.
- Detected face boxes are visible by default and remain in telemetry while fresh
  so users can tell whether recognition is working.
- Double-click enrollment saves the same face crop shown in the dialog as the
  person's thumbnail immediately; the worker may add an embedding crop later when
  face recognition succeeds.
- Clicked-face selection must prefer the face under the click. A body-click
  inside a person box must prefer a detected face in that track's expected head
  region instead of nearest-body distance; no-click fallback remains
  deterministic.

## Update Integrity Gates

Every published installer asset must be covered by the same release checksum
manifest that ships with the GitHub Release:

- `SHA256SUMS` is generated after all release artifact jobs have completed.
- Intel macOS DMGs are required release artifacts alongside Apple Silicon macOS,
  Windows, and Linux. If any required artifact job fails, the release is not
  published.
- The in-app updater must refuse a checksum mismatch or a selected asset missing
  from the checksum file.

## Source Viability

Every source used by Mark or release validation must prove:

- Frames arrive before the run starts.
- Timestamps or observed capture times advance.
- Dimensions are positive and stable enough for the selected benchmark.
- Pixel format is known or explicitly recorded.
- Invalid sources fail loudly before scoring starts.

NDI is the primary repeatable test path because fake streams can be generated,
but the source contract must remain generic for USB, RTSP, ONVIF, and synthetic
sources.

Fake NDI Mark runs must resolve every generated sender to the full advertised
NDI source name before registering cameras. Falling back to unresolved short
names is a failed preflight, not a valid benchmark.
